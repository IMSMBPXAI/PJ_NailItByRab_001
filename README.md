1. บทนำ (Introduction)
   
   1.1 บริบทของปัญหาและความสำคัญ (Background and Significance)
ในปัจจุบัน ธุรกิจการทำเล็บเจล เติบโตอย่างรวดเร็ว แต่กระบวนการเข้าถึงบริการยังมีอุปสรรคหลายประการ ทั้งประสบปัญหาเรื่องความยุ่งยากในการนัดหมาย ลูกค้าการขาดข้อมูลประกอบการตัดสินใจในการเลือกใช้บริการ จนต้องเสียเวลาทักแชทสอบถามคิวและรอแอดมินตอบกลับ ซึ่งอาจล่าช้าและสื่อสารคลาดเคลื่อน นอกจากนี้ร้านมักพบปัญหาการจัดการคิว เช่น การรับคิวซ้อน (Double Booking) หรือการประเมินเวลาทำเล็บคลาดเคลื่อน ลูกค้ายังขาดข้อมูลประกอบการตัดสินใจ เช่น หารีวิวหรือผลงานของช่างได้ยาก โครงการนี้จึงจัดทำขึ้นเพื่อพัฒนาแพลตฟอร์มดิจิทัลที่ช่วยแก้ปัญหาเหล่านี้เพื่อที่ลดข้อผิดพลาดในการทำงานและอำนวยความสะดวกให้ทั้งช่างทำเล็บและลูกค้า

   1.2 วัตถุประสงค์ของโครงการ (Objectives)
เพื่อพัฒนาแพลตฟอร์มดิจิทัลสำหรับการจองคิวร้านทำเล็บที่มีประสิทธิภาพ
เพื่อพัฒนาระบบบริหารจัดการคิวและตารางเวลาสำหรับร้านค้าและช่างทำเล็บ
เพื่ออำนวยความสะดวกให้นิสิตและบุคลากรมหาวิทยาลัยพะเยา รวมถึงบุคคลทั่วไป ให้เข้าถึงบริการได้ง่ายขึ้น

   1.3 ขอบเขตของระบบ (Scope)
ระบบนี้เป็นเว็บแอปพลิเคชันที่รองรับผู้ใช้งาน 2 กลุ่มหลัก คือ ลูกค้า (Customer) และ ช่างทำเล็บ (Nail Artist) เจ้าของร้าน (Admin) ระบบครอบคลุมตั้งแต่การสมัครสมาชิก การค้นหาบริการและแกลเลอรี การจองคิวพร้อมคำนวณระยะเวลาอัตโนมัติ ระบบชำระเงินมัดจำผ่าน QR Code รวมถึงการจัดการตารางงานและการรีวิวหลังรับบริการ



2. ภาพรวมของระบบ (Overall Description)
  
   2.1 บริบทการทำงานปัจจุบันและปัญหา (Current Context & Pain Points)
จากการลงพื้นที่สังเกตการณ์ ณ ร้าน NAIL IT BY RABBITZ เมื่อวันที่ 18 มกราคม 2569  พบปัญหาคอขวด (Bottlenecks) และความยุ่งยากในการทำงานดังนี้:การกระจายของข้อมูล: รับจองผ่านหลายช่องทาง (Facebook/LINE) ทำให้แอดมินตอบช้าหรือข้อมูลตกหล่น ปัญหาการจัดการคิว (Double Booking): การบันทึกลงสมุดจดหรือปฏิทินตั้งโต๊ะทำให้เกิดการจดซ้อนกัน หรือจัดคิวพนักงานพาร์ทไทม์ (นักศึกษา) ได้ยาก การประเมินเวลาผิดพลาด: ช่างมักลืมคำนวณเวลาบริการเสริม (เช่น ล้างเล็บเก่าใช้เวลาเพิ่ม 20 นาที - 1 ชม.) ทำให้คิวถัดไปต้องรอนาน การลืมนัดหมาย (No-show): ขาดระบบแจ้งเตือนลูกค้าล่วงหน้า

  2.2 กลุ่มผู้ใช้งาน (User Classes and Characteristics) ระบบออกแบบมาสำหรับผู้ใช้งาน 2 ประเภทหลัก ได้แก่:
ลูกค้า (Customer): ผู้รับบริการที่ต้องการค้นหาร้าน ลายเล็บ ดูรีวิว และจองคิวพร้อมระบุเวลา
ช่างทำเล็บ/เจ้าของร้าน (Nail Artist / Salon Owner / Admin): ผู้ให้บริการที่ต้องการสร้างโปรไฟล์ร้าน อัปโหลดผลงาน และจัดการตารางคิวรับลูกค้า

  2.3 ข้อมูลสภาพแวดล้อมและข้อจำกัด (Environment & Constraints)
ต้องมีการออกแบบ Responsive Design (เน้น Mobile-first) เพื่อให้แสดงผลตารางจองและรูปภาพได้สวยงามบนสมาร์ตโฟน
ระบบต้องสามารถรองรับตรรกะการคำนวณเวลา (Duration) ของการทำเล็บแต่ละแบบที่ไม่เท่ากันได้ เพื่อจัดสรรเวลาคิวได้อย่างแม่นยำ

  3.2 ความต้องการที่ไม่ใช่ฟังก์ชัน (Non-Functional Requirements - NFR)
    NFR-01: ประสิทธิภาพ (Performance): ระบบต้องสามารถโหลดแกลเลอรีรูปภาพและอัปเดตสถานะคิวว่างแบบ Real-time ได้อย่างรวดเร็ว เพื่อป้องกันผู้ใช้รายอื่นกดจองช่วงเวลาเดียวกัน (Double Booking)
    NFR-02: ความง่ายในการใช้งาน (Usability): UI ต้องมีความสวยงาม สอดคล้องกับธุรกิจความงามและรองรับ Responsive Design สมบูรณ์แบบ โดยเฉพาะบนหน้าจอมือถือ (Mobile-Friendly)
    NFR-03: ความน่าเชื่อถือ (Reliability): ข้อมูลการจองทั้งหมดต้องถูกรวมศูนย์ (Centralized Database)) เพื่อป้องกันปัญหาข้อมูลตกหล่นจากการจองผ่านหลายช่องทาง เครื่องมือที่ประมวลผลตรรกะการคำนวณระยะเวลา (Duration) อย่างแม่นยำ
    NFR-04: ความปลอดภัย (Security): รหัสผ่านของผู้ใช้งานต้องได้รับการเข้ารหัส (Hashing) ก่อนจัดเก็บลงฐานข้อมูล และต้องจำกัดสิทธิ์การตั้งค่าร้านค้าให้เฉพาะ Admin เท่านั้น
  
  3.3 กฎของธุรกิจ (Business Rules)
นโยบายการมัดจำ (Deposit Policy): ลูกค้าจะต้องชำระเงินมัดจำ 100 บาท ผ่าน QR Code ภายในเวลา 15 นาที เพื่อยืนยันการจองคิว การเก็บมัดจำนี้ใช้เป็นตัวคัดกรองลูกค้าที่ตั้งใจจะมาใช้บริการจริงๆ เพื่อป้องกันปัญหาการจองเล่น หากไม่ชำระเงินภายในเวลาที่กำหนด ระบบจะทำการยกเลิกใบจองนั้นโดยอัตโนมัติ
นโยบายการยกเลิกและการคืนเงิน (Cancellation Policy): หากลูกค้ายกเลิกการนัดหมายด้วยตนเอง ระบบจะไม่มีการคืนเงินสดให้ แต่จะเก็บยอดมัดจำนั้นไว้เพื่อใช้เป็นส่วนลดในรอบการจองถัดไป หรือหากไม่มีการจองเลยฏ็ไม่มีการคืนเงินทั้งสิ้น 
การล่วงเวลา (Lateness Policy): ร้านอนุญาตให้ลูกค้ามาสายจากเวลานัดหมายได้ไม่เกิน 15 นาที หากเกินจากเวลาดังกล่าว อาจส่งผลกระทบต่อคิวถัดไป ระบบและร้านค้าสามารถดำเนินการปรับเปลี่ยนคิวได้ตามความเหมาะสม หรืออาจพิจารณายกเลิกคิวของลูกค้ารายนั้น

  3.4 ตัวชี้วัดความสำเร็จ (Key Success Indicators - KSI)
ระยะสั้น: เอกสาร SRS สมบูรณ์ 100% และงาน Front-end ได้รับคะแนนประเมินด้าน UI/UX ในระดับดีจากทางร้าน และลูกค้า 
ระยะยาว: มีอัตราการใช้บริการ ทั้งการจองสำเร็จและการยกเลิกการจอง (Booking Rate) มากกว่าหรือเท่ากับ 10% จากยอดเข้าชมหรือยอดผู้ใช้งาน และมีผู้ใช้งานทั้งพนักงานร้านและลูกค้า สมัครใช้งานเพิ่มขึ้นอย่างต่อเนื่อง



4. ข้อกำหนดอื่นๆ

  4.1 ข้อกำหนดของฐานข้อมูล (Database Requirements)
ฐานข้อมูลต้องถูกออกแบบในลักษณะระบบรวมศูนย์ (Centralized Database) เพื่อเก็บรวบรวมข้อมูลทั้งหมดไว้ที่เดียว ป้องกันปัญหาข้อมูลตกหล่นจากการที่แอดมินรับจองผ่านหลายช่องทาง
ฐานข้อมูลต้องสามารถรองรับการจัดเก็บประเภทบริการที่หลากหลาย (เช่น เพ้นท์นิ้วมือ PVC กาวเจล กาวธรรมดา) รวมถึงต้องรองรับสถานะการจองที่จำเป็นต้องมีการอัปเดตตลอดเวลา
ต้องมีการเข้ารหัส (Hashing) รหัสผ่านของผู้ใช้งานทุกคนก่อนทำการจัดเก็บลงในฐานข้อมูล เพื่อความปลอดภัย

  4.2 ข้อกำหนดทางกฎหมายและข้อบังคับ (Legal and Regulatory Requirements)
ระบบต้องมีการจัดการสิทธิ์การเข้าถึงข้อมูลที่รัดกุม โดยต้องจำกัดสิทธิ์ในการตั้งค่าร้านค้าและการเข้าถึงข้อมูลเชิงลึก ให้ทำได้เฉพาะผู้ใช้งานในระดับ Admin เท่านั้น
ระบบต้องเก็บข้อมูลส่วนบุคคลและข้อมูลการชำระเงินของลูกค้า (สลิปโอนเงิน) อย่างปลอดภัยตามมาตรฐานสากล

  4.3 การทำให้เป็นสากลและการปรับให้เข้ากับท้องถิ่น (Internationalization and Localization)
ระบบแสดงผลด้วยภาษาไทยเป็นหลัก เพื่ออำนวยความสะดวกและสอดคล้องกับกลุ่มเป้าหมายหลัก ซึ่งได้แก่ นิสิตและบุคลากรของมหาวิทยาลัยพะเยา รวมถึงบุคคลทั่วไปในพื้นที่
ระบบใช้สกุลเงิน บาท (THB) ในการแสดงราคาประเมินและการเรียกเก็บเงินมัดจำ 100 บาท

  4.4 การจัดการความเสี่ยง (Risk Management / FMEA Matrix)
ความเสี่ยงด้านเทคนิค (Technical Risk): ระบบอาจไม่สามารถโหลดแกลเลอรีรูปภาพหรืออัปเดตสถานะคิวว่างแบบ Real-time ได้ทันท่วงที ซึ่งอาจนำไปสู่ปัญหาผู้ใช้งานหลายคนกดจองในช่วงเวลาเดียวกัน (Double Booking) หากมีลูกค้าทักมาพร้อมกัน
ความเสี่ยงด้านการดำเนินงาน (Operational Risk): ระบบอาจคำนวณระยะเวลา (Duration) คลาดเคลื่อน เนื่องจากบริการเสริมบางประเภท (เช่น การล้างเล็บเก่า หรือการต่อเล็บเพิ่ม) อาจต้องใช้เวลาบวกเพิ่มอีก 20 นาที ถึง 1 ชั่วโมง ทำให้คิวถัดไปต้องรอนาน นอกจากนี้พนักงานพาร์ทไทม์ที่เป็นนักศึกษาอาจมีตารางเรียนไม่แน่นอน ทำให้ระบบต้องยืดหยุ่นพอที่จะจัดสรรคิวช่างได้
ความเสี่ยงด้านผู้ใช้งาน (User Risk): ลูกค้าอาจลืมนัดหมาย (No-show) หากขาดระบบการส่งข้อความแจ้งเตือนเมื่อใกล้ถึงเวลา


5. ภาคผนวก

  5.1 คำจำกัดความและคำย่อ (Glossary)
    Double Booking (การจองคิวซ้อน): ปัญหาที่เกิดขึ้นเมื่อมีการบันทึกคิวของลูกค้ามากกว่า 1 รายลงในช่วงเวลาเดียวกัน หรือเกิดจากการที่ลูกค้าหลายคนตกลงโอนมัดจำเข้ามาพร้อมกัน
    No-show: เหตุการณ์ที่ลูกค้าทำการจองคิวไว้แล้วแต่ไม่มาปรากฏตัวตามนัดหมาย (ลืมนัด) ซึ่งทำให้ร้านสูญเสียรายได้
    Duration (ระยะเวลาการรับบริการ): ตรรกะการคำนวณเวลาที่ใช้ในการทำเล็บ ซึ่งแต่ละรูปแบบและประเภทบริการจะใช้เวลาไม่เท่ากัน ระบบจำเป็นต้องคำนวณเพื่อใช้จัดสรรเวลาคิวให้แม่นยำ
    Portfolio (แกลเลอรีผลงาน): รูปภาพผลงานจริงของช่างทำเล็บแต่ละคน ที่ถูกอัปโหลดขึ้นระบบเพื่อให้ลูกค้าใช้ดูเป็นข้อมูลประกอบการตัดสินใจก่อนจอง
    Responsive Design (Mobile-Friendly): การออกแบบหน้าเว็บไซต์ (UI) ให้สามารถปรับตัวและแสดงผลตารางคิวรวมถึงรูปภาพได้อย่างสวยงาม สมบูรณ์แบบเมื่อใช้งานบนหน้าจอมาร์ตโฟน
    Centralized Database (ฐานข้อมูลรวมศูนย์): การรวมศูนย์ข้อมูลการจองทั้งหมดไว้ในที่เดียว เพื่อแก้ปัญหาข้อมูลตกหล่นจากการที่แอดมินรับจองลูกค้าผ่านหลายช่องทาง (เช่น Facebook หรือ LINE)
